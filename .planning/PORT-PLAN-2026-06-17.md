## 2. Port candidates by subsystem

### A. Access layer (`internal/access/`) ← insane-search v0.5.0  — TOP PRIORITY

We already have `cascade_waf.go`, `phase4_tls.go`, `escalation.go`, 5 phases. Upstream matured the WAF
handling from a binary signal into a **profile system**. These are *enhancements to existing code*, not
greenfield.

| # | Feature | Status | Prio | Note |
|---|---------|--------|------|------|
| A1 | **WAF product profiles** (7 vendors: Akamai/Cloudflare/F5/AWS WAF/DataDome/PerimeterX/unknown) with confidence-ranked detection | `[NEEDS-VERIFY]` — we have `cascade_waf` but likely binary, not ranked profiles | P1 | Replace binary WAF signal with ranked `[]ProfileHit`; drives A2/A3 |
| A2 | **4-layer validation** (challenge markers → body-size → cookie state `_abck=~-1~` → CSS success selector) | `[NEEDS-VERIFY]` — phase4 checks 200 + JS challenge; depth unknown | P1 | Stricter "is this really a real page" gate; reduces false-success |
| A3 | **Per-WAF TLS avoid-list** (e.g. Akamai avoids chrome107/123/firefox utls fingerprints) | `[NEEDS-VERIFY]` — `phase4_tls.go` rotates utls; avoid-list unknown | P1 | Cheap win — skip known-blocked fingerprints; data-only addition |
| A4 | **bias-check linter** (no hardcoded site/brand names in access engine, CI-enforced) | New | P2 | Small Go pre-commit/`go test` lint; prevents site-specific drift |
| A5 | **R7 API-first parallel branch** (on WAF + collect-intent: probe internal `/api`/`/graphql`/`.json` in parallel) | New | P2 | Larger orchestration change in `cascade.go`; defer |

> insane-search is THE source for this layer and it's the one repo that genuinely leapt ahead. A1–A3

## 3. Recommended sequencing

1. **P1 batch — access layer (A1→A2→A3)**: one SPEC, "insane-search v0.5 profile port". Biggest gap,
   single subsystem, the three items chain (profiles feed validation + TLS avoid-lists). Start by
   reading `cascade_waf.go` + `phase4_tls.go` to confirm exactly what's binary vs ranked today.
2. **P1 singles (independent, parallelizable)**: B1 GitHub `search_commits` (confirmed gap, self-
   contained), C1 json_repair (confirmed gap, self-contained). No dependency on each other or on (1).
3. **P1 verify-then-port**: B2 X retry, C2 embedder truncation — read the file first; port only if the
   guard is actually absent.
4. **P2 cleanups**: A4 bias linter, D2 engine allow-list, C4/C3 observability, B3/B4/B5 minor guards,
   D1 only if usage check shows control-flow dependence.

All `[NEEDS-VERIFY]` items get a 1-file read before a SPEC line is written — per the project rule that
draft SPECs must not cite gaps that don't exist.
